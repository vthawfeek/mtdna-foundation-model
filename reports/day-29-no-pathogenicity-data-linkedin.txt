I built a pathogenic variant predictor for mitochondrial DNA. The architecture is sound. The training code works. The model weights are on HuggingFace.

I never built the evaluation dataset.

The model — MtDNAForVariantPathogenicity — uses the hidden state at the variant position rather than the [CLS] token. Point mutation at position 3,243? Take the embedding of the token covering that position, run it through a linear head, predict pathogenic/benign. That's the right inductive bias for a local property like pathogenicity.

Training pipeline: LoRA r=4 on the pre-trained encoder, binary cross-entropy with pos_weight=2.5 to handle class imbalance, a PathogenicityVariantDataset class that reads a parquet with (sequence, position, label) columns. This all runs. The model trains.

The evaluation parquet should contain:
- Pathogenic: ClinVar "pathogenic/likely pathogenic" mtDNA variants
- Benign: gnomAD common mtDNA variants (AF > 0.01)

The download clients for both databases exist in the pipeline. The join logic exists. But actually running it, validating labels against the reference sequence coordinates, and creating a clean held-out test split wasn't done. No data → no AUROC.

The lesson: define your evaluation benchmark before you write the model code. I designed the architecture thinking "ClinVar + gnomAD will be straightforward to download." That turned out to be true — the download code exists. What I didn't account for was the label validation, coordinate mapping, and split design that turns raw variant records into a clean binary classification benchmark. That's a full day of work that kept getting deprioritised.

The architecture is there. The gap is a data engineering task, not a modeling one.
