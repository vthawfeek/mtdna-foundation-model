I trained a foundation model on a laptop CPU. Here's what that actually looks like in numbers — including the deadlock that wasted 13.5 hours.

**The compute math:**
82,355 training windows / batch size 128 = 644 batches per epoch. On CPU, each forward+backward pass through a 6-layer, 256-dim transformer takes ~30 seconds. One epoch = ~6.5 hours. Two epochs = ~20 hours wall clock. Fine-tuning loss after 2 epochs: 3.266 vs random baseline ln(26) = 3.258. The model barely moved.

**The deadlock:**
I added num_workers=4 to the PyTorch DataLoader to parallelise data loading. On Linux, DataLoader spawns workers via fork(). Forked processes inherit the parent's thread state, including any locked mutexes from PyTorch's internal threads. The workers deadlocked trying to acquire those locks. Main process blocked waiting for batches that never arrived.

Visual evidence: 5 processes in sleeping state at 0% CPU. Total RAM consumed: ~20GB (3.9GB per worker copy of the dataset + model state). Duration: 13.5 hours before I killed it.

Fix: num_workers=0. Data loading in the main process, no forking, no deadlock. The additional cost: essentially nothing, because data loading was never the bottleneck — the transformer arithmetic was.

**The gap GPU would close:**
Same 2-epoch run: ~6 minutes on an A100. The 50-epoch run the LoRA actually needs to converge: ~25 minutes. On CPU: 425 hours. The model's failure to converge is a compute statement, not a model quality statement — the zero-shot k-NN at ~50% is the real measure of what the pre-training learned.

**Practical takeaways for CPU training:**
- Pre-tokenize to disk: avoids rebuilding 82k windows from scratch on every restart (57 seconds)
- num_workers=0 on Linux unless you've confirmed fork-safety
- Gradient checkpointing: halves peak memory at the cost of one extra forward pass
- Profile before optimising: on this setup, data loading was 0.2s per batch; the matrix operations were 29.8s. Don't parallelise the fast part.
