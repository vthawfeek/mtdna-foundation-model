I trained a foundation model on a laptop. Here's what CPU limits actually look like in numbers. 🧵
---
The dataset: 82,355 windows from 1,267 genomes after tokenization.
Batch size: 128
Batches per epoch: 644
Time per batch (CPU, 6-layer transformer, 256 hidden dim): ~30 seconds
Time per epoch: ~6.5 hours
Two epochs: ~20 hours wall clock
---
The DataLoader bug I hit:

Set num_workers=4 to speed up data loading. On Linux, DataLoader uses fork() by default. Forked workers inherit PyTorch thread mutexes from the parent process. Deadlock.

The training appeared to start — "Class weights computed" logged, MLflow run created. Then silence. For 13.5 hours.

All 5 processes (main + 4 workers): sleeping at 0% CPU. 19.5GB of RAM, doing nothing.

Fix: num_workers=0. No forking, no deadlock. Data loaded in the main process.
---
Memory: each DataLoader worker gets a full copy of the dataset.

82,355 windows × 512 tokens × 4 bytes = ~160MB per copy. But the full process (model weights + optimizer state + activations) uses ~3.9GB per worker.

Four workers: 15.6GB + 4.4GB main process = 20GB for a stuck process.

With num_workers=0: 4.4GB for the whole thing.
---
What GPU would change:

Training throughput: 100–200× faster. Each batch in milliseconds, not 30 seconds.

Same 2-epoch run: ~6 minutes on an A100 instead of 20 hours.

50 epochs (what the LoRA actually needs to converge): ~25 minutes on GPU, 425 hours on CPU.

The model didn't converge in 2 CPU epochs. On GPU it would have converged before lunch.
---
What you CAN do on CPU:

- Pre-tokenize and cache all windows to disk (save the 57-second rebuild on restart)
- Use gradient checkpointing (halves memory at the cost of one extra forward pass)
- Smaller batch size with gradient accumulation (same effective batch, less RAM per step)
- Profile where the 30 seconds goes: forward pass ~18s, backward pass ~12s, data loading ~0.2s

The data loading is not the bottleneck. The matrix multiplications are.
